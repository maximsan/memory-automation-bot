Project Onboarding

Goal: in about 15 minutes, know how to setup the bot, trace one capture, and choose the right file for the next change.

1. Mental Model

This project is not a Notion app, not an n8n workflow, and not a general assistant. Version A is a small Next.js API service that uses Telegram for capture/review and Notion for durable state.

Telegram Capture surface and review buttons.
Notion Durable source of truth: Projects, Notes, Tasks.
OpenAI Extracts summary, project guess, and at most one task.
Vercel Queue Moves slow AI/media work out of the webhook path.

2. Setup Path

This is the canonical setup guide. Other docs should point here instead of repeating these steps.

Local files

  1. Install dependencies: npm install.
  2. Copy env template: cp .env.example .env.local.
  3. Keep real values only in .env.local and Vercel environment variables. Do not put real tokens in .env.example.

Telegram

  1. Create a bot with BotFather.
  2. Set TELEGRAM_BOT_TOKEN in .env.local.
  3. Send a fresh message to the bot from your Telegram account.
  4. Before setting a webhook, run curl "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/getUpdates".
  5. Use the numeric message.from.id as TELEGRAM_ALLOWED_USER_IDS.

If getUpdates returns an empty result, send a new message to the bot and try again. If a webhook was already set, remove it first with deleteWebhook or use the user ID you already know.

Notion

  1. Create a Notion internal integration.
  2. Use the integration secret as NOTION_TOKEN.
  3. Create a parent page named Project Memory Bot. A private page is enough for this personal version.
  4. Share that parent page with the integration.
  5. Copy the page ID from the Notion page URL and set it as NOTION_PARENT_PAGE_ID.

The setup script creates or reuses the Projects, Notes, and Tasks databases under that parent page.

OpenAI and cron

  1. Set OPENAI_API_KEY.
  2. Set CRON_SECRET to any long random value. It is just the bearer secret for the weekly cron route.
  3. The weekly reminder uses one Vercel cron job. It is intentionally approximate; the requirement is once per week.
  4. Model and limit variables can stay at the values from .env.example unless you intentionally change them.

Seed projects and Notion databases

  1. Optional seed only: copy config/projects.example.json to config/projects.json if you want setup to create first project rows.
  2. Skip the seed file if you prefer to add projects later from Telegram with /addproject.
  3. Run npm run setup:notion. The script loads .env.local automatically.

Vercel and GitHub

  1. Connect the GitHub repository to Vercel.
  2. Add the same runtime secrets from .env.local to the Vercel project environment variables.
  3. GitHub Actions only runs checks from .github/workflows/checks.yml. It does not deploy and does not need Vercel deploy secrets.
  4. Vercel Git integration creates preview deployments for pull requests and production deployments from main.
  5. For the safest simple flow, enable GitHub branch protection on main and require the Checks workflow before merge.

First deployment and webhook

  1. Push a branch and open a pull request.
  2. Wait for GitHub Checks.
  3. Open the Vercel preview URL and verify /api/health.
  4. Merge to main.
  5. Open the Vercel production URL and verify /api/health.
  6. Set the Telegram webhook: curl "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/setWebhook?url=https://YOUR_PROJECT.vercel.app/api/telegram/webhook".
  7. Send a simple Telegram text capture and approve it.

Local cron test

Use ?force=1 only for local manual cron testing. It bypasses the weekly schedule and bearer token on localhost, but production rejects forced cron runs.

Local vs Deployed

Local development is for fixtures, type checks, prompt edits, and unit tests. Vercel deployment is needed for the real Telegram webhook and queue worker. GitHub Actions does not deploy this project; it only runs checks. Vercel Git integration owns preview and production deployments.

3. Development Loop

npm run lint
npm run typecheck
npm test
npm run build

Use the full loop before deployment-facing changes. GitHub runs the same checks in .github/workflows/checks.yml. For a tiny prompt-only edit, start with focused tests, then run the full loop before opening a pull request.

4. Trace One Capture

  1. app/api/telegram/webhook/route.ts loads config and concrete integrations.
  2. src/routes/webhook.ts dispatches command, callback, summary edit, or capture.
  3. src/core/telegram.ts parses raw Telegram update payloads into a small internal union.
  4. src/routes/webhook.ts creates a Notion note before queueing, so Notion is the durable state.
  5. app/api/queues/process-capture/route.ts consumes queue jobs.
  6. src/core/processCapture.ts calls OpenAI, matches project, updates the note, and edits the Telegram review message.
  7. src/core/reviewActions.ts handles approval, project state updates, and optional task creation.

5. Where To Edit First

  • Command behavior: src/routes/commands.ts.
  • Review text/buttons: src/core/format.ts.
  • AI extraction prompt: prompts/extract-note.md.
  • AI project state prompt: prompts/update-project-state.md.
  • Notion schema: update docs/product-spec.md first, then setup/store code.
  • Queue or cron: route under app/api, config in vercel.json.
