Project Onboarding

Goal: in about 15 minutes, know how to setup the bot, trace one capture, and choose the right file for the next change.

1. Mental Model

This project is not a Notion app, not an n8n workflow, and not a general assistant. Version A is a small Next.js API service that uses Telegram for capture/review and Notion for durable state.

Telegram Capture surface and review buttons.
Notion Durable source of truth: Projects, Notes, Tasks.
OpenAI Extracts summary, project guess, and at most one task.
Vercel Queue Moves slow AI/media work out of the webhook path.

2. Setup Path

  1. Install dependencies: npm install.
  2. Copy env template: cp .env.example .env.local.
  3. Create a Telegram bot with BotFather and set TELEGRAM_BOT_TOKEN.
  4. Send a message to your bot, then run curl "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/getUpdates" before a webhook is set. Use the numeric message.from.id for TELEGRAM_ALLOWED_USER_IDS.
  5. Create a Notion internal integration, create a parent page named Project Memory Bot, share the page with the integration, and set NOTION_TOKEN plus NOTION_PARENT_PAGE_ID.
  6. Add OPENAI_API_KEY and CRON_SECRET.
  7. Optional: copy config/projects.example.json to config/projects.json.
  8. Run npm run setup:notion to create or reuse the Notion databases.
  9. Deploy to Vercel and copy the production URL.
  10. Set the Telegram webhook: curl "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/setWebhook?url=https://YOUR_PROJECT.vercel.app/api/telegram/webhook".

Primary setup docs live in README.md, product-spec.md, and version-a-vercel.md.

Local vs Deployed

Local development is for fixtures, type checks, prompt edits, and unit tests. The real end-to-end bot is deployment-first because Telegram needs a public HTTPS webhook and the queue worker is Vercel-owned.

3. Development Loop

npm run lint
npm run typecheck
npm test
npm run build

Use the full loop before deployment-facing changes. For a tiny prompt-only edit, start with focused tests, then run the full loop before deploying.

4. Trace One Capture

  1. app/api/telegram/webhook/route.ts loads config and concrete integrations.
  2. src/routes/webhook.ts dispatches command, callback, summary edit, or capture.
  3. src/core/telegram.ts parses raw Telegram update payloads into a small internal union.
  4. src/routes/webhook.ts creates a Notion note before queueing, so Notion is the durable state.
  5. app/api/queues/process-capture/route.ts consumes queue jobs.
  6. src/core/processCapture.ts calls OpenAI, matches project, updates the note, and edits the Telegram review message.
  7. src/core/reviewActions.ts handles approval, project state updates, and optional task creation.

5. Where To Edit First

  • Command behavior: src/routes/commands.ts.
  • Review text/buttons: src/core/format.ts.
  • AI extraction prompt: prompts/extract-note.md.
  • AI project state prompt: prompts/update-project-state.md.
  • Notion schema: update docs/product-spec.md first, then setup/store code.
  • Queue or cron: route under app/api, config in vercel.json.

6. Retrieval Check

Answer from memory before looking back. This is the part that builds durable understanding.

Primary Source To Read

Read the official Next.js route handler docs when you want to understand why the API entrypoints live in app/api/**/route.ts. For this project, route handlers are the outer shell; the important behavior should still stay in testable modules under src.
